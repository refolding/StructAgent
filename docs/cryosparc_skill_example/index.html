TL;DR

The cryoSPARC skill is a self-contained skill package in StructAgent. It is written as a practical example for Claude Code / Codex-style coding agents, but the design is runtime-agnostic: any agent can adopt it if it can load a skill instruction file, read reference files on demand, and run approved shell/Python tools.

What this is

The skill covers cryoSPARC single-particle-analysis workflows: import, preprocessing, picking, extraction/2D, ab initio, homogeneous/heterogeneous/non-uniform refinement, 3D classification, 3DVA/3DFlex, local/focused refinement, masks, symmetry, helical processing, CryoSPARC Live, cryosparc-tools, cryosparcm, GPU lanes/queues, storage, RELION interop, troubleshooting, and error lookup.

This page is not a benchmark or a claim that an agent should autonomously run cryoSPARC. It is an implementation example: how to package domain knowledge, keep site-specific state separate, and require explicit confirmation before queueing jobs or touching shared compute.

Safety boundary

Credentials, hostnames, project IDs, private datasets, and raw cryoSPARC outputs are intentionally not included. Live automation requires the local user to provide access and approve actions. The bundled helper defaults to dry-run/read-only behavior.

The agent loads SKILL.md, then only the relevant references — for example 09_local_refinement.md, 20_masks.md, and orientation_and_preferred_views.md — rather than loading the whole corpus.

Why this works well for Claude/Codex

The skill is plain files plus a Python helper. A coding agent can inspect the files, run scripts, patch site_config.md, and preserve changes in git. There is no hidden server component.

Minimum agent requirements

The agent should be able to read local files, run Python/shell tools, keep secrets out of generated code, and ask for explicit human confirmation before queueing GPU jobs, deleting data, restarting services, or changing cluster/lane configuration.

Make it site-aware

A generic cryoSPARC skill becomes useful only after the agent learns the local deployment. Keep those facts in a local, non-public file such as site_config.md next to SKILL.md.

# ~/.claude/skills/cryosparc/site_config.md
# Local file. Do not commit credentials.

## cryoSPARC instance
- master_url: http://<cryosparc-host>:<base-port>
- master_version: v5.0.x
- cryosparc_tools_python: ~/.venvs/cs-tools-v5/bin/python
- default_lane: <gpu-lane-name>

## Access rules
- Prefer read-only inspection first.
- Queue only after explicit user confirmation of project_uid, workspace_uid, lane, and dry-run vs real queue.
- Never store passwords or API/session tokens here.

Typical probes the agent can run locally:

ProbeWhy
python --version, pip show cryosparc-toolsConfirm client version and Python compatibility.
cryosparcm status or admin-provided versionConfirm master version when the user has admin access.
read-only cryosparc-tools connection testSeparate connectivity/version issues from credential issues.
inspect parent job resourcesDiscover the actual lane before queueing follow-up jobs.

Use on a live cryoSPARC instance

The safe progression is:

  1. Advisor mode first. Identify the workflow stage and load the smallest relevant reference file.
  2. Read-only inspection. Confirm version, project/workspace/job IDs, job type, inputs, outputs, params, status, and lane.
  3. Dry-run plan. Show the exact jobs/params/connections that would be created.
  4. Human confirmation. Require explicit project_uid, workspace_uid, lane, and dry-run vs queue.
  5. Queue only when confirmed. Record job IDs and report what changed.

The included harness enforces that queueing is not accidental:

$ python skills/annika/cryosparc/scripts/cryosparc_harness.py create-job \
    --project-uid P1 --workspace-uid W1 --job-type new_local_refine \
    --params-json '{"use_alignment_prior": true}'
# default: dry-run JSON only; no job is queued

$ python skills/annika/cryosparc/scripts/cryosparc_harness.py create-job \
    --project-uid P1 --workspace-uid W1 --job-type new_local_refine \
    --lane gpu --commit --queue --queue-confirm QUEUE
# real queue only with explicit commit + queue + lane + confirmation word

Never commit these

Passwords, cryoSPARC session tokens, private hostnames if they should not be public, private project titles, raw maps, particle stacks, downloaded job assets, or screenshots containing unpublished dataset details.

Reusable implementation patterns

1 · Version parity before auth debugging

If cryosparc-tools cannot connect, compare client and master versions before assuming bad credentials. Major cryoSPARC API changes can make old clients fail before authentication.

2 · Load references narrowly

For a local-refinement question, load 09_local_refinement.md and maybe 20_masks.md; for an exact error string, start with 17_error_lookup.md and 15_troubleshooting.md.

3 · Separate public skill from private site config

The GitHub repo contains reusable instructions and references. Each lab/user creates local site_config.md and project memory after install.

4 · Inspect the parent job before copying it

Do not guess lane, inputs, or parameter names. Read them from the existing job and from ui_to_api_crosswalk.md.

5 · Queueing is a privileged action

GPU jobs consume shared resources and can modify project state. The agent should ask before queue/start/delete/admin operations.

Snippets

Connection skeleton

import os
from cryosparc.tools import CryoSPARC

cs = CryoSPARC(
    host=os.environ["CRYOSPARC_HOST"],
    base_port=int(os.environ.get("CRYOSPARC_BASE_PORT", "39000")),
    email=os.environ["CRYOSPARC_EMAIL"],
    password=os.environ["CRYOSPARC_PASSWORD"],
)
assert cs.test_connection()
